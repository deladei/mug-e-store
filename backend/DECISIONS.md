| 2026-06-13 | Redemption discount is **capped at the subtotal** (points never offset the delivery fee), and an over-cap request is **capped, not rejected** — only the points that produce a discount are spent, the surplus stays on the balance. Rate: **1 point = 1 pesewa** (PRD: 100 points = GHS 1). Over-**balance** requests ARE rejected (`409 insufficient_points`). | Plan §3 explicitly left the cap target to be decided + documented. Capping at subtotal is symmetric with earning (points are earned on subtotal), keeps the delivery fee a real cost, and guarantees `total ≥ delivery_fee ≥ 0`. Owner chose subtotal-only + cap. The rule lives in pure `domain.Redemption` (unit-tested); `store.Checkout` reads the balance and writes the negative ledger row inside the checkout transaction under a `SELECT … FOR UPDATE` lock on the user row to prevent double-spend. |
| 2026-06-13 | Cancelling a redeemed order writes a compensating **`refund_on_cancel`** positive ledger entry (in `TransitionOrder`, inside the same transaction as the cancel). | PRD §S1 requires redeemed points to be returned if an order is cancelled before completion. Without it, checkout's `cancelDangling` (fired when Paystack init fails) would silently burn the customer's points on an order that never even reached payment. Cancelled is terminal and the no-op guard short-circuits a repeat cancel, so the refund is written at most once. Not in plan §3's explicit list, but required for correctness — flagged here rather than left as a silent gap. |
| 2026-06-13 | Adding `points_to_redeem` to `POST /checkout` is a **public API shape change** (CLAUDE.md §9). | The API consumption brief previously told the frontend "redemption is not yet a backend feature — do not build the control." Building it was authorized by the owner; the brief §2.5/§2.8 are updated, and the contract change must be mirrored to the monorepo via the PR flow (Git Law §3.5). |
| 2026-06-14 | Phase 2 **S3 password reset** built backend-only via two endpoints — `POST /auth/password-reset/request {email}` (always `200`, generic body) and `POST /auth/password-reset/confirm {token, password}`. New table `password_reset_tokens` (migration `0002`), modelled on `refresh_tokens`: **only the SHA-256 hash is stored** (`auth.HashToken`), tokens are **single-use** (deleted on confirm) and **expire after 1 hour**. A successful reset **revokes all of the user's refresh tokens** (a password change ends every session) — done atomically in `store.ConsumePasswordReset` (one txn: validate-under-`FOR UPDATE` → update password → delete all the user's reset tokens → delete all their refresh tokens). | PRD §S3. New endpoints = additive public-API-shape change (CLAUDE.md §9); published to API brief §2.9. The hashed-secret + single-use + expiry design reuses the refresh-token security model already defended in this repo. Session-revocation-on-reset is standard hygiene (a leaked-then-reset account must not keep live sessions) and is the reason confirm is transactional. |
| 2026-06-14 | **No email provider is wired** for password reset. The request endpoint logs the reset link server-side (`slog.Info`, with a `TODO(deploy)` marker) and **never returns the raw token in the HTTP response**; the link is built from `FRONTEND_ORIGIN` (no new env var). | CLAUDE.md §5 forbids adding dependencies (an SMTP/email client) without approval, and the PRD itself flags S3 as stretch precisely *because* "it requires email sending." Email transport is therefore an owner/infra step — like the Paystack live-E2E and deploy items in `STATE.md` — and the backend half (token lifecycle + endpoints + security) is what's deliverable in-sandbox. Returning the token in the response was rejected outright: it would let anyone reset any account, defeating the whole model. Production swaps the single log call for a real mailer to `u.Email`. |
| 2026-06-14 | Password-reset **request never reveals account existence**: a registered email, an unknown email, and a malformed email all return the same `200` + generic message; the confirm endpoint collapses unknown / already-used / expired tokens into one `400 invalid_token`. | CLAUDE.md §4 (no account enumeration) — the same rule that makes login return identical errors for unknown-email vs wrong-password. Both reset endpoints are under the existing per-IP auth rate limiter (10/min). |
| 2026-06-13 | Phase 2 **S2 admin reports** implemented as `GET /admin/reports/summary?days=` returning `{from,to,totals,daily[]}`. Design choices: **revenue = orders whose status is NOT `pending_payment` or `cancelled`** (money actually taken); **`daily` is continuous** (gap days zero-filled, for charting); window default 30 days, clamped 1–365; days grouped on `created_at::date` with **no timezone conversion** (the shop is in Ghana = GMT/UTC+0); **admin-only** (revenue is financial data, stricter than the staff-or-admin queue endpoints). | The PRD lists `GET /admin/reports/summary (Phase 2)` and "orders/day, revenue/day" but no response shape or revenue definition; this fills those in. New endpoint = additive API-shape change (CLAUDE.md §9); published to the API brief §4.1. Chosen as the next in-sandbox deliverable because S3 (password reset) needs email infra and S4 (guest checkout) is frontend-led — S2 is backend-only and self-contained. |
| 2026-06-18 | **Deploy-on-merge is CI-gated, not Render-native autoDeploy.** `render.yaml` sets `autoDeploy: false`; a `.github/workflows/deploy.yml` listens for the **CI** workflow to finish (`workflow_run`), and only when it concluded `success` on `main` does it POST a Render **Deploy Hook** (`secrets.RENDER_DEPLOY_HOOK_URL`). The job fails fast with a clear message if the secret is unset. | Plain Render autoDeploy ships every push to `main` regardless of test outcome — a broken build could reach production. Routing the deploy through `workflow_run` after CI means prod only ever receives a commit whose build+vet+test+govulncheck+coverage all passed. The Deploy Hook URL is the only secret and lives in GitHub Actions secrets, never in the repo. Escape hatch documented: flip `autoDeploy: true` + disable the workflow to fall back to push-deploys. | 
| 2026-06-18 | **Render deploy via `render.yaml` Blueprint** (Docker web service + managed Postgres). The service uses `runtime: docker` (`dockerfilePath: ./Dockerfile`), `healthCheckPath: /api/v1/healthz`, and a `preDeployCommand` that applies `migrations/*.up.sql` via the image's psql before each release. `DATABASE_URL` is wired `fromDatabase`; `JWT_SECRET` is `generateValue: true`; `PAYSTACK_SECRET_KEY` and `FRONTEND_ORIGIN` are `sync: false` (dashboard-only secrets, never in the blueprint). `PORT` is omitted so Render injects it (config defaults to 8080 only when unset). | Plan §6 leaves hosting to the owner; a checked-in Blueprint makes it a one-click provision instead of manual dashboard clicking, and documents the exact env contract. Reused the existing Dockerfile rather than Render's native Go builder so local `docker compose` and prod build the same artifact. Two free-tier foot-guns are flagged in-file: Render **free Postgres expires ~90 days** (the AESIS project was bitten by exactly this — use `basic-256mb`+ for real), and **free web services can't run `preDeployCommand`** (run migrations once from the Render Shell instead). Secrets are `sync:false` so a blueprint re-apply never clears them. | 
| 2026-06-18 | **Security/production-readiness hardening pass, triaged to scope.** Added: (1) a `securityHeaders` middleware (`nosniff`, `X-Frame-Options: DENY`, `Referrer-Policy: no-referrer`, CSP `default-src 'none'`, CORP, HSTS) wired into the global chain + unit test; (2) a GitHub Actions CI workflow (`.github/workflows/ci.yml`) running build/vet/test against a real Postgres service, a **govulncheck** dependency scan, and a **40% coverage floor** (regression guard); (3) the Dockerfile build stage pinned to `golang:1.22.12-alpine` (reproducible, carries stdlib CVE fixes); (4) `docs/production-readiness.md` (full checklist triage) + `docs/architecture.md` (layering / FSM / payment-flow diagrams). | A reviewer's generic prod checklist (multi-tenancy, GDPR/HIPAA, chaos engineering, circuit breakers, caching, RTO/RPO, …) was assessed against this system's real scope (single-tenant, one shop, viva-graded). The senior-engineer call is **fit-to-scope**, not cargo-culting: implement the controls that match the risk (headers, dep scanning, CI gates) and **explicitly record the deliberate non-goals** (multi-tenancy is a PRD §2 non-goal; HIPAA is N/A — no health data; chaos/circuit-breakers/caching add failure modes without reducing real risk here) so the viva has crisp answers rather than silent gaps. `govulncheck` found 33 stdlib CVEs, **all** from the old local `go1.22.4` toolchain and **none** in our own call paths; the fix is building on a current 1.22.x (CI + the pinned Docker base), so no code change was warranted. Coverage floor set to 40% as a backslide guard — the correctness-critical `internal/domain` is already 100%. |

Verified end-to-end (`docker compose up --wait`): migrate applied 0001–0003, seed wrote 2 staff + 4/8/14, `api` went healthy, `/healthz`→`{"status":"ok"}` and `/categories` returned the seeded menu. | Plan §6 ("host backend + Postgres") is owner-gated only at the *hosting* step; the container definition itself needs no secrets and is buildable in-sandbox, so it is the right increment to do now — it makes deploy a one-image push to any container host (Render/Railway/Fly). Docker is infrastructure, not a pinned-stack dependency (CLAUDE.md §5), and adds no Go module. Compose env values are clearly dev-only (`sk_test_placeholder`, throwaway `JWT_SECRET`); production supplies real secrets out of band. The only owner choices left are the platform and the real Paystack/JWT secrets. |
| 2026-06-18 | **Demo seeding is a Go program (`cmd/seed`), not the `seed.sql` the schema doc names.** `make migrate-up && make seed` loads an 8-item / 14-variant Ghanaian Accra menu (all integer pesewas) plus two demo accounts `admin@coffeemug.shop` / `staff@coffeemug.shop` (password `password123`, roles admin/staff). It reads only `DATABASE_URL`, is idempotent (skips the menu when the catalog is non-empty, skips each account when it already exists), and writes exclusively through `internal/store`. A `Makefile` (build/run/seed/test/vet/migrate-up/migrate-down) and a real README replaced the placeholders the implementation plan §2 assumed already existed. | Plan §2 (a `/docs` spec) calls `make migrate-up`, `make seed`, `make run` as the "immediate next step," but none existed — a code-vs-spec gap (CLAUDE.md §2). Chose a Go seeder over the schema doc's `seed.sql` because the seeded accounts need a **bcrypt** password hash: a static hash baked into SQL is brittle and opaque, whereas `cmd/seed` calls `auth.HashPassword` like the real register path and keeps all SQL inside `internal/store` (§4 layering). The menu leans local (sobolo, Ghana cocoa, bofrot) per CLAUDE.md §1 (the shop is in Accra). No new dependency; `migrate-up`/`migrate-down` shell out to `psql`. This unblocks the owner-only §2 run-locally, §4 Paystack E2E, and §6 deploy steps. |
| 2026-06-16 | Phase 2 **S4 guest checkout** modelled as a **passwordless `users` row**, not a nullable `orders.user_id`. New endpoint `POST /auth/guest {name?, phone?}` mints a guest: a real `users` row with `is_guest=true` (migration `0003`), a synthetic unique non-routable email (`guest-<token>@guest.coffeemug.local`), and an **unusable** bcrypt hash (of a random secret no one holds). It returns the standard login session, so cart / checkout / ownership / SSE / history all work **unchanged** for a guest. Guests **earn no loyalty** — `TransitionOrder` skips the `earn_on_completion` row when `is_guest`. `name`/`phone` are optional (anonymous guest → "Guest"); the checkout contract is untouched (order address/phone still come from `POST /checkout`). | PRD §S4. The "guest is a customer who can't log in" model means **zero** changes to the user_id-keyed write paths — a nullable `orders.user_id` would have forced every ownership/cart/SSE query and the FK to special-case NULL, multiplying risk across the system for one feature. The synthetic email keeps the `UNIQUE`/`NOT NULL` email contract intact with no schema change beyond the marker flag; the unusable hash means the constant-time login check can never succeed for a guest even if its email were guessed (no enumeration path either, CLAUDE.md §4). Suppressing loyalty avoids littering the append-only ledger with rows no one can ever read or spend. New endpoint = additive public-API-shape change (CLAUDE.md §9); published to API brief §2.4 and mirrored to the monorepo via the PR flow. |
